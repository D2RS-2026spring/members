## 存储备份策略



### 你的数据存放在哪里

**课程核心文件**（如课件、代码）：存放在本地电脑 **D 盘** 的 `course` 目录中。

**个人资料**：分散存放在本地电脑的 **C 盘** 与 **D 盘** 中。



### 如果电脑坏了，你能恢复多少数据？

系统故障：如果只是系统故障，可以尝试恢复本地硬盘中的文件，但恢复后的整理成本较高。
物理损坏或丢失：如果硬盘发生物理损坏或电脑丢失，且没有备份，本地数据将彻底丢失。
云端数据：只有已经提交到 GitHub 等平台的代码和作业才可能被恢复。



### 制定一个3-2-1备份方案

#### 核心原则

**3份数据总量**：保留三份完整的数据副本（1份原始数据 + 2份备份）。

**2种不同介质**：将数据存储在至少两种不同的存储介质上（如：本地NAS + 云存储）。

**1份异地备份**：至少有一份备份存放在物理距离较远的异地（如：云端或另一处房产），以防火灾、盗窃等物理灾难。

**1份离线/不可变备份**：至少有一份备份是离线（Air-gapped）或开启对象锁（不可篡改）的，以防止勒索病毒加密所有在线备份。

**0错误验证**：定期进行恢复演练，确保备份数据是可用的，错误率为零。



#### 架构分层设计

- 热数据层（工作层）

  **位置**：高性能SSD/NVMe本地硬盘。

  **用途**：日常高频读写。

  **策略**：不做冗余，依赖下一层的实时同步。

- 温数据层（本地冗余层）

  **位置**：本地NAS或DAS（配置RAID 5/6或RAID 10）。

  **用途**：实时同步热数据，提供快速恢复（分钟级）。

  **策略**：使用快照技术（如ZFS或Btrfs），每小时保留快照，防止误删除或文件损坏。

- 冷数据层（异地归档层）

  **位置**：公有云对象存储（如AWS S3 Glacier, 阿里云OSS归档型）或冷备服务器。

  **用途**：灾难恢复。

  **策略**：每日增量，每周全量。开启版本控制。
